# Dasi Agent 技术设计方案

## 一、项目概述

### 1.1 项目定位
Dasi Agent 是一个集成了 AI 对话、多角色 Agent 工作流、RAG 知识库和 MCP 工具调用的全栈 AI Agent 平台。项目采用前后端分离架构，提供用户端对话界面和管理后台配置界面。

### 1.2 核心价值
- **统一对话平台**：支持普通对话、RAG 增强对话、MCP 工具调用对话、Agent 工作流对话四种模式
- **可配置化**：所有配置（API、Model、MCP、Advisor、Prompt、Client、Agent）均可通过后台管理
- **可视化工作流**：基于 vue-flow 的流程画布，直观展示 Agent 与 Client 的连接关系
- **模块化设计**：基于 DDD 架构，职责清晰，易于扩展

### 1.3 技术栈

#### 前端技术栈
- **框架**：Vue 3 + Vite
- **UI 库**：TailwindCSS
- **状态管理**：Pinia
- **路由**：vue-router
- **流程图**：vue-flow（用于可视化工作流）
- **HTTP 客户端**：Axios

#### 后端技术栈
- **框架**：Spring Boot 3
- **JDK**：Java 17
- **AI 框架**：Spring AI（集成 OpenAI 兼容接口）
- **数据库**：MySQL（业务数据）+ PostgreSQL（向量存储）
- **缓存**：Redis
- **ORM**：MyBatis
- **容器化**：Docker

### 2.2 DDD 分层架构

  - `/api/v1/ai/work/execute`：Agent 工作流执行（SSE）
  - `/api/v1/ai/rag/file`：上传文件到 RAG 知识库
  - `/api/v1/ai/rag/git`：上传 Git 仓库到 RAG 知识库

- **AdminController**：管理后台 CRUD 接口
- **AuthController**：用户认证接口
- **QueryController**：查询接口
- **SessionController**：会话管理接口

#### 2.2.3 ai-agent-domain（领域层）

**管理服务模块**
- **AdminService**：管理后台业务逻辑

**会话服务模块**

**工具模块**
- **MessageService**：消息持久化


**关键实现**：

### 3.2 Agent 工作流系统

#### 3.2.1 工作流策略

项目支持两种工作流策略：

**Loop 策略（循环模式）**：
```
Analyzer（分析任务）
    ↓
Performer（执行任务，调用 MCP 工具）
    ↓
Supervisor（监督评估）
    ↓
通过？→ Summarizer（总结）
    ↓
完成
```
- **适用场景**：需要多轮迭代、监督评估的复杂任务
- **核心节点**：
  - Analyzer：分析用户需求，制定执行计划
  - Performer：执行具体任务，调用 MCP 工具
  - Supervisor：评估执行结果，决定是否继续
  - Summarizer：总结最终结果

**Step 策略（步骤模式）**：
```
Inspector（检查任务）
    ↓
Planner（规划步骤）
    ↓
Runner（执行步骤，支持重试）
    ↓
Replier（回复用户）
```
- **适用场景**：线性步骤、需要重试机制的任务
- **核心节点**：
  - Inspector：检查任务可行性
  - Planner：规划执行步骤
  - Runner：执行步骤（失败则重试 maxRetry 次）
  - Replier：整理结果并回复用户

#### 3.2.2 执行流程

**请求入口**：`AiController.execute()`

**流程**：
```java
1. 参数校验与状态校验
2. 会话权限校验
3. 构建 ExecuteRequestEntity
4. DispatchService.dispatchExecuteStrategy()
   - 根据 agentId 选择对应的执行策略（Loop/Step）
   - 提交到线程池异步执行
5. 通过 SSE 实时推送执行结果
6. 持久化消息与统计数据
```

**执行上下文传递**：
- ExecuteContext：在整个工作流中传递数据
- 支持多轮轮次（round）计数
- 每个节点的输出保存到 Context，供后续节点使用

**关键代码位置**：
- `DispatchService.java:81-106`
- `ExecuteLoopStrategy.java`
- `ExecuteStepStrategy.java`

#### 3.2.3 节点实现

每个节点继承自 `AbstractExecuteNode`，实现 `doApply()` 方法：

```java
@Override
protected String doApply(ExecuteRequestEntity request, ExecuteContext context) throws Exception {
    // 1. 从 Context 获取前置节点的输出
    String analyzerResponse = context.getValue(ANALYZER.getContextKey());

    // 2. 获取对应的 ChatClient
    ChatClient performerClient = getBean(clientBeanName);

    // 3. 构建提示词（支持模板变量替换）
    String performerPrompt = flowPrompt.formatted(
        context.getUserMessage(),
        analyzerResponse
    );

    // 4. 调用 AI 模型（带 ChatMemory）
    String performerResponse = performerClient
        .prompt(performerPrompt)
        .advisors(a -> a
            .param(CHAT_MEMORY_CONVERSATION_ID_KEY, sessionId)
            .param(CHAT_MEMORY_RETRIEVE_SIZE_KEY, CHAT_MEMORY_RETRIEVE_SIZE_WORK))
        .call()
        .content();

    // 5. 解析 JSON 结果
    JSONObject performerObject = parseJsonObject(performerResponse);

    // 6. 通过 SSE 推送分段结果
    ExecuteResponseEntity responseEntity = ExecuteResponseEntity.createPerformerResponse(...);
    sendSseMessage(context, responseEntity);

    // 7. 保存到 Context
    context.setValue(PERFORMER.getContextKey(), performerJson);

    // 8. 返回下一个节点
    return router(request, context);
}
```

**关键代码位置**：`ExecutePerformerNode.java`

```java
@Component
public class AuthInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        String uri = request.getRequestURI();
        if (uri.startsWith("/admin/")) {
            // 校验管理员权限
            UserContext user = getUserFromToken(request);
            if (!UserRole.ADMIN.getRole().equals(user.getRole())) {
                response.setStatus(403);
                return false;
            }
        }
        return true;
    }
}
```

## 五、技术亮点

### 5.1 Armory 自动装配机制

**问题**：如何在系统启动时自动装配 API、Model、MCP、Advisor、Client？

**解决方案**：
1. **配置驱动**：通过 armory 配置文件指定需要预装配的 clientId 和 agentId
2. **责任链模式**：使用 bugstack-wrench 框架的责任链树，按顺序装配各组件
3. **缓存防重**：使用 Redis Set 记录已装配的 ID，避免重复装配

**装配流程**：
```
API → Model → MCP → Advisor → Client → Flow → Agent
```

**关键代码**：`ArmoryConfig.java:48-66`

### 5.2 策略模式 + 责任链模式

**问题**：如何支持多种工作流策略（Loop/Step）？

**解决方案**：
1. **策略工厂**：ExecuteStrategyFactory 根据 agentId 获取对应的执行策略
2. **责任链节点**：每个节点继承 AbstractExecuteNode，实现 doApply() 方法
3. **上下文传递**：ExecuteContext 在整个责任链中传递数据

**优势**：
- 新增策略无需修改现有代码
- 每个节点职责单一，易于维护
- 支持灵活的节点组合

### 5.3 SSE 实时推送

**问题**：如何在 Agent 工作流执行过程中实时推送进度？

**解决方案**：
使用 Spring SseEmitter：
```java
SseEmitter sseEmitter = new SseEmitter(0L);

// 发送分段结果
ExecuteResponseEntity responseEntity = ExecuteResponseEntity.createPerformerResponse(...);
sseEmitter.send(SseEmitter.event()
    .name("performer")
    .data(responseEntity));

// 完成
sseEmitter.complete();
```

**前端监听**：
```javascript
const eventSource = new EventSource('/api/v1/ai/work/execute');
eventSource.addEventListener('performer', (event) => {
    const data = JSON.parse(event.data);
    // 渲染 Performer 节点输出
});
eventSource.addEventListener('complete', () => {
    eventSource.close();
});
```

### 5.4 动态 MCP 工具挂载

**问题**：如何在运行时动态挂载不同的 MCP 工具？

**解决方案**：
1. **动态创建 McpSyncClient**：根据 mcpIdList 创建对应的 MCP Client
2. **封装为 SyncMcpToolCallbackProvider**：统一挂载到 ChatClient
3. **自动初始化**：MCP Client 创建后自动 initialize()

**优势**：
- 按需调用，节省资源
- 支持同一 MCP 服务的多实例
- 失败隔离：单个 MCP 服务失败不影响其他

### 5.5 Redis 缓存 + AOP 切面

**问题**：如何优化频繁查询的配置数据？

**解决方案**：
1. **AOP 切面**：通过 @Cacheable 和 @CacheEvict 注解自动管理缓存
2. **Redis 存储**：使用 Redis 缓存热点数据
3. **自动失效**：配置更新时自动清除对应缓存

**示例**：

```java
public AiClientVO queryAiClientVOByClientId(String clientId) {
    // 查询数据库
}

@CacheEvict(key = "#clientId")
public void updateAiClient(AiClient client) {
    // 更新数据库，自动清除缓存
}
```
