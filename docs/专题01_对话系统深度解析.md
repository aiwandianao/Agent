## 一、真实业务场景

### 场景 1：智能客服问答

**业务需求**：

**对应技术**：

- RAG 增强对话 → 检索代码库

### 场景 3：个人助理

**业务需求**：
- 处理日常提醒
- 查询天气、行程

## 三、三种对话模式详解

### 3.1 普通对话模式

#### 业务场景
适用于简单的问答场景，不需要外部知识或工具调用。

#### 实现流程

```
用户输入 "你好"
    ↓
参数校验 (sessionId, message, clientId)
    ↓
会话校验 (验证用户权限)
    ↓
获取 ChatClient (根据 clientId 从 Spring 容器获取)
    ↓
设置 ChatMemory (sessionId 作为 conversationId)
    ↓
调用 LLM API
    ↓
返回结果 → 持久化 → 统计
```

#### 核心要点

| 要点 | 说明 |
|-----|------|
| **ChatClient 获取** | 按 clientId 从 Spring 容器获取，每个 Client 对应不同的模型配置 |
| **会话记忆** | 使用 `sessionId` 作为 `conversationId`，不同会话记忆隔离 |
| **参数控制** | temperature、presencePenalty、maxTokens 可由前端传入 |
| **流式支持** | 通过 `flux()` 实现流式返回 |

├── chatComplete()  (非流式对话)   行: 136-158
└── chatStream()     (流式对话)    行: 160-208

### 3.2 RAG 增强对话模式

#### 业务场景

**问题 1**：LLM 知识有截止日期，不知道最新信息
**问题 2**：LLM 不知道私有数据（如公司内部文档）
**解决方案**：RAG (Retrieval-Augmented Generation) 检索增强生成

#### 实现原理

```
用户输入 "公司年假政策是什么？"
    ↓
向量化用户输入
    ↓
向量检索 (检索知识库中相关文档)
    ↓
构建增强提示词：
    "以下是相关文档内容：[检索到的文档]
     请基于以上文档回答用户问题：公司年假政策是什么？"
    ↓
调用 LLM
    ↓
返回基于文档的准确答案
```

#### 关键组件

| 组件 | 作用 |
|-----|------|
| **向量数据库** | 存储文档的向量表示，支持相似度检索 |
| **Embedding 模型** | 将文本转换为向量（本项目使用 OpenAI embedding） |
| **检索器** | 根据用户输入检索相关文档 |
| **提示词模板** | 将检索结果注入到系统提示中 |

#### 核心实现

```java
// 1. 构建检索条件（按知识库标签过滤）
Filter.Expression expression = filterExpressionBuilder
    .eq("knowledge", ragTag)  // ragTag 是知识库标识
    .build();

// 2. 执行向量检索
SearchRequest searchRequest = SearchRequest.builder()
    .query(userMessage)
    .filterExpression(expression)
    .topK(5)  // 检索前 5 条最相关的文档
    .build();

List<Document> documents = pgVectorStore.similaritySearch(searchRequest);

// 3. 注入系统提示
String documentString = documents.stream()
    .map(Document::getText)
    .collect(Collectors.joining("\n"));

return List.of(
    new SystemPromptTemplate(RAG_SYSTEM_PROMPT)
        .createMessage(Map.of("documents", documentString)),
    new UserMessage(userMessage)
);
```

#### RAG 提示词模板示例

```
你是一个专业的客服助手。请基于以下文档内容回答用户问题：

【参考文档】
{documents}

【用户问题】
{question}

【要求】
1. 答案必须基于参考文档
2. 如果文档中没有相关信息，明确告知用户
3. 保持专业、友好的语气
```

### 3.3 MCP 工具调用对话模式

#### 业务场景

**问题**：LLM 是一个语言模型，只能"说"，不能"做"
**解决方案**：通过 MCP 协议赋予 LLM 调用外部工具的能力

#### 实现原理

```
用户输入 "帮我查一下北京今天的天气"
    ↓
LLM 理解意图，判断需要调用天气工具
    ↓
LLM 生成工具调用请求：
    {
      "tool": "checkWeather",
      "arguments": {"location": "北京"}
    }
    ↓
系统执行工具调用 (MCP Server)
    ↓
返回工具执行结果：
    {"temperature": "25°C", "weather": "晴"}
    ↓
LLM 基于工具结果生成最终回复
    ↓
"北京今天天气晴朗，温度 25 摄氏度"
```

#### MCP 通信方式

| 方式 | 适用场景 | 实现方式 |
|-----|---------|---------|
| **SSE** | 远程部署的 MCP 服务 | HTTP 长连接，服务端推送 |
| **STDIO** | 本地进程的 MCP 服务 | 标准输入输出通信 |

#### 动态工具挂载流程

```java
// 1. 根据 mcpIdList 动态创建 MCP Client
List<McpSyncClient> mcpClients = new ArrayList<>();
for (String mcpId : mcpIdList) {
    McpConfig mcpConfig = getMcpConfig(mcpId);

    // 创建 SSE 或 STDIO 连接
    McpSyncClient client = createMcpClient(mcpConfig);
    client.initialize();
    mcpClients.add(client);
}

// 2. 封装为 ToolCallbackProvider
SyncMcpToolCallbackProvider toolCallbacks =
    new SyncMcpToolCallbackProvider(mcpClients);

// 3. 挂载到 ChatClient
chatClient.prompt()
    .toolCallbacks(toolCallbacks)  // 挂载工具
    .call()
    .content();
```

#### 已实现的 MCP 工具

| 工具服务 | 功能 | 场景 |
|---------|------|------|
| mcp-server-amap | 天气查询 | "帮我查一下天气" |
| mcp-server-csdn | 发布文章到 CSDN | "帮我发布这篇文章" |
| mcp-server-wecom | 发送企业微信消息 | "通知团队成员开会" |
| mcp-server-email | 发送邮件 | "给客户发送邮件" |
| mcp-server-bocha | 联网搜索 | "搜索最新的 AI 新闻" |

## 四、会话记忆管理

### 4.1 为什么需要会话记忆？

**问题场景**：
```
用户: "我的订单什么时候发货？"
系统: "您的订单将在明天发货"
用户: "那退货运费谁承担？"  ← LLM 不知道"那"指什么
```

**解决方案**：维护会话历史，让 LLM 能够理解上下文。

### 4.2 ChatMemory 实现

```java
chatClient.prompt()
    .advisors(a -> a
        .param(CHAT_MEMORY_CONVERSATION_ID_KEY, sessionId)  // 会话 ID
        .param(CHAT_MEMORY_RETRIEVE_SIZE_KEY, 10))          // 保留最近 10 条
    .call()
    .content();
```

### 4.3 记忆存储策略

| 对话类型 | 记忆条数 | 原因 |
|---------|---------|------|
| **Chat 对话** | 10 条 | 普通对话较短，10 条足够 |
| **Work 工作流** | 50 条 | 工作流对话较长，需要更多上下文 |

### 4.4 存储位置

```
Redis (键值结构)
├── chat_memory:{sessionId}
│   ├── [{role: "user", content: "..."}]
│   ├── [{role: "assistant", content: "..."}]
│   └── ...
```

**选择 Redis 的原因**：
- 高性能读写
- 支持 TTL 自动过期
- Spring AI 原生支持

### 5.2 关键校验点

| 校验点 | 校验内容 | 失败处理 |
|-------|---------|---------|
| **参数校验** | sessionId, message, clientId 是否为空 | 返回 400 错误 |
| **会话校验** | 用户是否有权限访问该会话 | 返回 403 错误 |
| **Client 存在性** | clientId 对应的 ChatClient 是否存在 | 返回配置错误 |
| **MCP 可用性** | MCP 服务是否正常运行 | 降级处理，不挂载工具 |

#### 架构设计

**Q7**: 为什么要将对话入口分为 complete（非流式）和 stream（流式）？

<details>
<summary>点击查看答案</summary>

**答案**：
- **非流式 (complete)**：
  - 适用于需要完整结果后再处理的场景
  - 如：后台任务、批量处理
  - 实现简单，易于集成
- **流式 (stream)**：
  - 适用于需要实时反馈的场景
  - 如：聊天界面、长文本生成
  - 用户体验更好，但实现稍复杂

</details>

---

**Q8**: 增强服务（AugmentService）在对话链路中起什么作用？

<details>
<summary>点击查看答案</summary>

**答案**：
- **RAG 增强**：检索知识库，将结果注入系统提示
- **MCP 增强**：动态挂载工具到 ChatClient
- **职责单一**：只负责"增强"，不负责"调用"
- **灵活性**：可按需启用/禁用不同增强方式

</details>

---

**Q9**: 会话校验（Session Validation）的目的是什么？

<details>
<summary>点击查看答案</summary>

**答案**：
- **安全**：确保用户只能访问自己的会话
- **隔离**：不同用户的会话数据互不干扰
- **审计**：记录谁在什么时候访问了什么会话
- **实现方式**：通过 `sessionService.validateSessionAccess()` 校验

</details>

**Q10**: 对话系统如何保证性能？

<details>
<summary>点击查看答案</summary>

**答案**：
- **缓存**：配置数据缓存（AOP + Redis）
- **异步**：消息持久化和统计记录异步执行
- **连接池**：HTTP 客户端连接池复用
- **流式返回**：减少首字响应时间
- **数据库优化**：索引、分表、读写分离（根据规模）

</details>

**Q13**: 如何实现一个"多轮对话填写表单"的功能？

<details>
<summary>点击查看答案</summary>

**答案**：
**场景**：收集用户信息（姓名、电话、地址）

**实现方案**：
1. **启用 ChatMemory**：保留会话历史
2. **设计 Prompt**：
   ```
   你需要收集用户的以下信息：姓名、电话、地址
   请逐项询问，每次只问一个字段
   如果用户提供了信息，确认后继续下一项
   所有信息收集完成后，调用 submitForm 工具提交
   ```
3. **开发 MCP 工具**：
   ```java
   @Tool(description = "提交表单")
   public String submitForm(@P("姓名") String name,
                           @P("电话") String phone,
                           @P("地址") String address) {
       // 保存到数据库
       return "表单提交成功";
   }
   ```
4. **对话流程**：
   ```
   系统: 请问您的姓名？
   用户: 张三
   系统: 好的，您的电话？
   用户: 13800138000
   系统: 好的，您的收货地址？
   用户: 北京市朝阳区...
   系统: [调用 submitForm] 表单提交成功！
   ```

</details>
